# prototype

## Scope

HTML interactive prototype or clickable product flow used to review screens, states, and user tasks.

## Standard Sections

1. App frame or prototype shell
2. Primary screen or starting state
3. Navigation, tabs, or step controls
4. Core task state
5. Secondary state, empty state, or error state
6. Completion or confirmation state

## Required Inputs

- User role
- Task flow
- Key screens or states
- Interaction depth
- Device targets

## Design System Hints

- Prioritize realistic UI structure over marketing copy.
- Use familiar controls for navigation, forms, tabs, filters, and actions.
- Represent state changes clearly even when interactivity is lightweight.
- Keep component dimensions stable so state changes do not shift layout unexpectedly.

## Variation Axes

- Flow depth
- Fidelity level
- Navigation model
- State coverage
- Desktop versus mobile emphasis

## Review Fields

- User task
- Screen list
- Interaction model
- State coverage
- Fidelity level

## QC Focus

- Prototype screens and states are visible and reviewable.
- Primary task flow is understandable without external explanation.
- Controls look like actual controls, not marketing CTAs.
- Mobile or desktop framing matches requested device targets.
- Any JavaScript is minimal, local, and non-dangerous.
